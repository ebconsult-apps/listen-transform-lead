# Component findings

The 48 domain components (`src/components/**`) + the visually-defining shadcn/ui primitives, audited as 7 clusters.

---

## Site chrome & shared marketing sections

> Site chrome and shared marketing sections for the CLEAR / Erik Bohjort site. The bones are reasonable: the header is a real <header> with a fixed scroll-shrink state, NavLink is used so an .active class is applied automatically, the footer is semantic (<footer>, <address>, real © year), TrustedBy logos use loading="lazy", BookPreview's modal traps Escape and locks body scroll, CookieConsent persists consent and gates GA4, and SEO.tsx is genuinely solid (canonical, OG, Twitter, JSON-LD, noindex support). The problems are concentrated in three areas: (1) interaction-state gaps on the custom marketing controls (.btn-primary / .btn-secondary / .nav-link have hover but NO :active and NO visible focus ring — only the unused shadcn ui/button has focus-visible), and the nav active state is visually identical to hover so there is no real "you are here"; (2) textbook AI-slop component patterns — a 3-card testimonial carousel with dots, a 37-logo grayscale trusted-by wall with empty alt, a 4-column-style footer link farm duplicating the whole nav, and dead href="#" Privacy/Terms; (3) CTA duplication — the homepage stacks three near-identical "assessment" CTAs (Hero, an inline Assessment section, and CTASection). Typography leans on the documented Inter-everywhere problem and makes it worse by routing headings through SF Pro Display, a system font that is never loaded and is absent on non-Apple devices. There is also a real CSS bug: the Hero background gradient uses rgba(var(--primary)...) against an HSL space-separated token, so the hero's only background texture silently renders nothing.

**16 findings** — 5 high · 9 medium · 2 low

- **🔴 HIGH · Accessibility** — The custom marketing controls used throughout the chrome — .btn-primary, .btn-secondary, and .nav-link — have NO visible focus indicator. index.css defines hover (.nav-link:hover:after, .btn-primary:hover:before) but no :focus-visible/:focus styles, and none of the components add focus utilities. Only the shadcn ui/button.tsx has focus-visible, and the chrome never uses it. Keyboard users tabbing the header nav, Hero CTAs, footer links, and CTA buttons get no on-screen focus ring.
  - **Evidence:** src/index.css:81-111 (.nav-link / .btn-primary / .btn-secondary define hover only, no focus-visible); src/components/Layout.tsx:57-77 (nav-link); src/components/Hero.tsx:159-166 (btn-primary/btn-secondary)
  - **Fix:** Add a shared focus-visible ring to the component layer, e.g. .btn-primary, .btn-secondary, .nav-link { @apply focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2; }. Verify it is visible on both the transparent and white scrolled header states.
  - **Rule:** _Missing focus ring (visible focus indicator is an accessibility requirement)_
- **🔴 HIGH · Interactivity & States** — No :active / pressed feedback on any chrome button. .btn-primary animates a hover sheen (:before scale-x) and .btn-secondary only shifts bg on hover; neither has an :active state, so clicks feel non-physical. This is the documented global .btn-primary gap, and the chrome reproduces it on every CTA (Hero, CTASection, BookPreview, ServicesPreview footer CTA, Index inline Assessment).
  - **Evidence:** src/index.css:97-111 (.btn-primary / .btn-secondary — no :active); reused at src/components/CTASection.tsx:56-62, src/components/BookPreview.tsx:90-99, src/components/ServicesPreview.tsx:118-121
  - **Fix:** Add a pressed state to the shared button classes: .btn-primary, .btn-secondary { @apply active:scale-[0.98] active:translate-y-px; } with the existing transition-all duration so the click reads as a physical press.
  - **Rule:** _No active/pressed feedback (add scale(0.98)/translateY on press)_
- **🔴 HIGH · Interactivity & States** — The current-page nav indicator is visually identical to hover, so there is no real "you are here." .nav-link:hover:after and .nav-link.active:after both apply the exact same w-1/2 underline in the same primary color. On any page, the active link is indistinguishable from a hovered link, and with the pointer away from the bar the active page reads no differently from the six inactive links.
  - **Evidence:** src/index.css:85-95 (.nav-link:hover:after { @apply w-1/2 } and .nav-link.active:after { @apply w-1/2 } are identical); used in src/components/Layout.tsx:69-77
  - **Fix:** Differentiate the active state: give .nav-link.active a persistent full-width or thicker underline plus a weight/color bump (e.g. @apply text-foreground font-medium with a w-full bottom border), distinct from the hover half-underline, and add aria-current="page" via NavLink's render-prop.
  - **Rule:** _No indication of current page in navigation_
- **🔴 HIGH · Component Patterns** — TestimonialsSection is the exact 3-card carousel-with-dots testimonial pattern the audit calls out as generic AI. It rotates one quote at a time via prev/next chevrons plus a row of dot indicators, over a hand-rolled opacity-juggling fade. The skill explicitly says to replace this with a masonry wall, embedded social posts, or a single rotating quote.
  - **Evidence:** src/components/TestimonialsSection.tsx:119-150 (ChevronLeft/ChevronRight controls + dot map with active dot widening to w-8)
  - **Fix:** Drop the carousel chrome. Show all three testimonials at once as an asymmetric quote wall (varied widths), or commit to a single strong rotating quote with no dots. If kept as a carousel, at minimum it needs pause-on-hover, keyboard arrow support, and aria-live so screen readers announce slide changes.
  - **Rule:** _3-card carousel testimonials with dots_
- **🔴 HIGH · Strategic Omissions** — Footer Privacy and Terms links are dead — both point to href="#". Beyond being dead links, these are the legally-meaningful links a consulting site collecting leads and running GA4 (with a cookie banner) is expected to honor, so they cannot scroll-to-top placeholders.
  - **Evidence:** src/components/Layout.tsx:132-133 (<a href="#">Privacy</a>, <a href="#">Terms</a>)
  - **Fix:** Point Privacy and Terms at real routes/pages. If the pages do not exist yet, create them — the CookieConsent banner and GA4 integration make a real privacy policy effectively mandatory.
  - **Rule:** _Dead links / No legal links_
- **🟠 MED · Content** — Testimonial authors are anonymized to job-title-only with generic category companies — "CEO / International Healthcare Provider", "CEO / Deep Tech Startup", "CCO / Housing Organisation". With no real names, no faces, and two of three authors literally labeled "CEO", the social proof reads as placeholder and is low-trust for a consulting brand selling on credibility.
  - **Evidence:** src/components/TestimonialsSection.tsx:6-25 (author: "CEO"/"CCO", company: "International Healthcare Provider" etc.)
  - **Fix:** Use real attributable quotes with name, title, and organization (with permission), or replace with logo-backed mini case results. If anonymity is contractually required, at least vary the titles and add sector/region specificity so it doesn't read as a template.
  - **Rule:** _Generic names / placeholder company names (social proof must feel real)_
- **🟠 MED · Component Patterns** — The footer is a link farm that duplicates the entire primary navigation. All seven nav destinations (Home, About, Resources, Services, Methodology, Get the Book, Contact) are re-listed under a "Navigation" column that simply mirrors the header. The skill says to simplify footers to main paths plus legally-required links rather than re-printing the whole nav.
  - **Evidence:** src/components/Layout.tsx:102-113 (footer "Navigation" column re-lists the same 7 NavLink targets from lines 69-77)
  - **Fix:** Trim the footer to a few high-intent links (e.g. Contact, Get the Book, the primary service page) plus legal, and let the header own full navigation. Avoid duplicating the complete menu verbatim.
  - **Rule:** _Footer link farm with 4 columns_
- **🟠 MED · Layout** — The homepage stacks three near-duplicate assessment CTAs, violating the 'no two CTAs with the same intent on a page' gate. Hero: 'Take the Free Assessment'. Inline Assessment section: 'How Ready Is Your Organization for Change?' / 'Start the Assessment'. CTASection: 'Ready to Break the Cycle of Failed Change?' / 'Take the Free Assessment'. All three drive to /assessment, and two share the identical label 'Take the Free Assessment'. Hero and CTASection additionally repeat the same secondary 'Book a Discovery Call'.
  - **Evidence:** src/pages/Index.tsx:51-77 (Hero -> inline Assessment section:67 -> CTASection:77); src/components/Hero.tsx:159 vs src/components/CTASection.tsx:57 (duplicate label 'Take the Free Assessment')
  - **Fix:** Pick one assessment CTA per page (the bottom CTASection is the natural closer). Remove the mid-page inline Assessment block or repurpose it to a different next-step, and ensure no two buttons on the page share the same label/intent.
  - **Rule:** _CTA labels: no two CTAs with the same intent on a page_
- **🟠 MED · Typography** — Headings have no real typographic presence because 'display' resolves to SF Pro Display, a font that is never loaded and is absent on every non-Apple device — falling back to Inter var. Only Inter is @imported (index.css:end), and the body is also Inter var, so on Windows/Android every heading and paragraph render in the identical typeface. This is the documented Inter-everywhere problem made worse by depending on an un-shipped system font for the one element meant to differ.
  - **Evidence:** tailwind.config.ts:69-70 (sans: ['Inter var',...], display: ['SF Pro Display','Inter var',...]); src/index.css imports only rsms.me/inter (no SF Pro / display webfont loaded)
  - **Fix:** Self-host or load an actual display typeface with character (e.g. Outfit, Cabinet Grotesk, or a serif for an editorial consulting feel) instead of relying on SF Pro Display, so headings are visually distinct from Inter body on all platforms. At minimum, tighten heading tracking/leading so they read as display type.
  - **Rule:** _Browser default fonts or Inter everywhere / Headlines lack presence_
- **🟠 MED · Code Quality** — The Hero's only background texture silently renders nothing due to a malformed color. The radial gradients use rgba(var(--primary, 0,0,255), 0.15), but --primary is defined as HSL space-separated channels '220 80% 48%', so the value expands to rgba(220 80% 48%, 0.15) which is invalid CSS and is dropped. The decorative hero backdrop the code intends to fade in is effectively invisible, leaving the hero a flat plain panel.
  - **Evidence:** src/components/Hero.tsx:120-121 (radial-gradient(... rgba(var(--primary, 0, 0, 255), 0.15) ...)); token defined at src/index.css:17 (--primary: 220 80% 48%)
  - **Fix:** Use the project's HSL convention: hsl(var(--primary) / 0.15) (and 0.1), matching how the rest of the codebase consumes tokens, so the hero gradient actually renders. Then verify the opacity-20 wrapper still reads as subtle.
  - **Rule:** _Empty, flat sections with no visual depth / import-and-token correctness_
- **🟠 MED · Component Patterns** — The 'Trusted by' section is a generic 37-logo wall rendered at grayscale + opacity-40, and every logo has alt="" aria-hidden="true", so the trust signal is invisible to screen readers and assistive tech sees nothing. A wall of 37 faded uniform marks reads as filler rather than curated proof, and the data array carries a stale 'logo-31 (removed)' comment.
  - **Evidence:** src/components/TrustedBy.tsx:76-87 (grayscale opacity-40, alt="" aria-hidden="true"); stale comment at TrustedBy.tsx:3
  - **Fix:** Curate to a smaller set of recognizable logos, give each a real alt (the org name) so the trust signal is accessible (or label the group with a visible heading and keep individual marks decorative only if names are duplicated elsewhere), and lift the opacity slightly for legibility. Remove the dead 'logo-31 (removed)' comment.
  - **Rule:** _Missing alt text on images / Commented-out dead code_
- **🟠 MED · Code Quality** — CookieConsent is entirely inline-styled and hardcodes zIndex: 9999, sitting outside the rest of the project's Tailwind token system and z-index scale. It also has no focus-visible ring and no :active state on its two buttons, and the 'Decline' ghost button uses a 1px border at rgba(255,255,255,0.4) on a dark panel which is faint.
  - **Evidence:** src/components/CookieConsent.tsx:60-118 (full inline style block, zIndex: 9999, Decline border rgba(255,255,255,0.4))
  - **Fix:** Move the banner to Tailwind classes and a named z-index from the scale (the header already uses z-50), and add focus-visible + active states to both buttons. Strengthen the Decline button outline (or give it a subtle filled surface) so it clears UI contrast.
  - **Rule:** _Inline styles mixed with CSS classes / Arbitrary z-index values like 9999_
- **🟠 MED · Color & Surfaces** — The CLEAR-phase accent palette is leaking into LewinModel's shared section, breaking the one-locked-accent rule. LewinModel hardwires blue-500/amber-500/green-500 for its three stages (text, backgrounds, dots, progress bar), introducing three saturated Tailwind hues alongside the site's single primary (hsl 220 80% 48%). This is the multi-color CLEAR-phase risk the brief flags, using raw Tailwind palette colors rather than the defined --phase-* tokens.
  - **Evidence:** src/components/LewinModel.tsx:9-30 (color: 'text-blue-500'/'text-amber-500'/'text-green-500', bgColor); LewinModel.tsx:116,126,144,152 (blue/amber/green chips, headings, dots, bar)
  - **Fix:** Either keep this section on the single locked primary accent, or, if per-stage color is intentional, drive it from the existing --phase-* tokens (index.css:48-52) so the hues are calibrated to the brand rather than stock blue/amber/green. Don't mix raw Tailwind-500 colors into an otherwise single-accent page.
  - **Rule:** _Color-consistency lock: one accent locked across the whole page (5-color CLEAR phase palette)_
- **🟠 MED · Component Patterns** — Two different home sections render nearly the same component for two different frameworks, and both even share the same DOM id. FrameworkPreview (CLEAR, 7 steps) and LewinModel (Unfreeze/Change/Refreeze, 3 stages) are the identical left-list / right-detail-card / progress-bar / auto-advance pattern, and both set id="framework-preview" — duplicate ids if ever rendered together, and a copy-paste tell. Both also auto-advance on a timer with no pause-on-hover/focus, which fights a user reading a step.
  - **Evidence:** src/components/FrameworkPreview.tsx:101 (id="framework-preview", auto-advance:92-98) vs src/components/LewinModel.tsx:84 (same id="framework-preview", auto-advance:68-77)
  - **Fix:** Extract a single shared stepper component parameterized by data and color token, and give each instance a unique id. Add pause-on-hover/focus (and respect prefers-reduced-motion) to the auto-advance so it doesn't reset while someone is reading.
  - **Rule:** _Generic card look / repeated cliche pattern (DRY shared sections) + No active/pressed and motion that ignores reduced-motion_
- **🟡 LOW · Iconography** — Icons are exclusively Lucide across all chrome and sections (Menu/X, ArrowRight, ExternalLink, Quote, ChevronLeft/Right, BookOpen, plus the cliche metaphor set ShieldCheck for safety/trust and FlaskConical/Compass/Heart). Lucide-only is the default AI icon fingerprint per the audit, and ShieldCheck-for-safety is a called-out cliche metaphor.
  - **Evidence:** src/components/Layout.tsx:4; src/components/Hero.tsx:3; src/components/TestimonialsSection.tsx:3; src/components/FrameworkPreview.tsx:4 (ShieldCheck, Compass, Heart); src/components/ServicesPreview.tsx:4 (FlaskConical)
  - **Fix:** Either swap to a less-default set (Phosphor/Heroicons) for differentiation, or keep Lucide but audit the metaphors — replace shield-for-safety and similar literal icons with less obvious choices, and standardize stroke widths.
  - **Rule:** _Lucide or Feather icons exclusively / cliche metaphors (shield for security)_
- **🟡 LOW · Content** — AI-cliche subtitle copy and an unfinished section. ServicesPreview's CTA reads 'Explore Our Approach' and the FrameworkPreview intro leans on softened marketing phrasing, while LewinModel ships with an empty <h2> 'The Model of Change' whose intro paragraph and a header control area were deleted, leaving blank/whitespace-only markup. The em-dash is used in UI copy in several spots (BookPreview body, Hero subtitles), which the gate bans for UI copy.
  - **Evidence:** src/components/LewinModel.tsx:88-92 (heading with empty/removed paragraph) and 120-123 (empty flex control div); src/components/BookPreview.tsx:80 ('fail — even when') and Hero subtitles (e.g. Hero.tsx:33 'on paper.' variant uses ' — ')
  - **Fix:** Fill or remove the empty LewinModel intro/control blocks rather than shipping blank nodes, tighten 'Explore Our Approach' to a concrete next step, and replace UI-copy em-dashes with commas/periods or restructured sentences per the em-dash ban.
  - **Rule:** _AI copywriting cliches / Em-dash ban in UI copy_

## Marketing templates & lead-capture forms

> This cluster of marketing templates and lead-capture forms is functionally solid: forms POST through a shared useFormSubmit hook with real inline error banners (no window.alert), the hook is defensively coded against the SPA-rewrite false-200 trap, LeadForm does proper Zod field validation with per-field messages and clear-on-change, and DictationButton is genuinely well-built (real aria-pressed, aria-live interim transcript, supported/unsupported states, hover states, EN/SV toggle). However there are several real, code-grounded problems. (1) The big one: success copy across every form violates the exclamation-mark ban and uses a generic 'Success!' toast. (2) The hand-rolled form inputs in LeadForm and WhitepaperGate ship a weaker, inconsistent focus ring (focus:ring-primary/30, no ring-offset) than the project's own .input utility and shadcn Input (focus:ring-ring + offset) — a visible focus-quality and consistency regression. (3) Those same inputs set no placeholder color, falling back to the browser default instead of the design system's placeholder:text-muted-foreground. (4) No button has an :active/pressed state (global .btn-primary gap, exercised heavily here). (5) Validation is uneven: LeadForm validates with Zod, but FreeChapterForm and WhitepaperGate rely only on native required, so the user gets native browser tooltips instead of the inline error pattern used elsewhere. (6) WhitepaperGate's newsletter 'checkbox' is a plain button with icon glyphs and no role=checkbox/aria-checked, so it is invisible-as-a-checkbox to screen readers. (7) NichePage hides all body content at opacity-0 and only reveals it via JS setTimeout, so a JS failure leaves the page blank. (8) CalendlyEmbed is misnamed dead-comment cruft — it is a Microsoft Bookings link, not a Calendly embed, and not an embed at all. (9) NichePage uses the textbook one-filled + one-ghost CTA pair with Title Case heading.

**12 findings** — 3 high · 6 medium · 3 low

- **🔴 HIGH · Content** — Every form's success message uses an exclamation mark, which the checklist explicitly bans ('Exclamation marks in success messages. Remove them.'). LeadForm toast 'Consultation request sent!', FreeChapterForm toast 'Your chapter is ready!' plus inline heading 'Thank You!', and WhitepaperGate toast title is the bare generic 'Success!' with description 'Your whitepaper is ready to download.' 'Success!' as a title is doubly weak — it's both an exclamation and a content-free label.
  - **Evidence:** LeadForm.tsx:97 ('Consultation request sent!'), FreeChapterForm.tsx:33 ('Your chapter is ready!') and FreeChapterForm.tsx:46 ('Thank You!'), WhitepaperGate.tsx:46-47 (title 'Success!')
  - **Fix:** Drop the exclamation marks and make titles informative: LeadForm -> title 'Request received', desc unchanged. FreeChapterForm -> 'Your chapter is ready' and inline heading 'Your chapter is ready' (not 'Thank You!'). WhitepaperGate -> replace 'Success!' with 'Your whitepaper is ready' as the title. Confident, not loud.
  - **Rule:** _Content: Exclamation marks in success messages_
- **🔴 HIGH · Interactivity & States** — The hand-rolled inputs/selects/textarea in LeadForm and WhitepaperGate use focus:ring-primary/30 — a 30%-opacity focus ring on a white card — and omit ring-offset. The project's own design-system controls use a full-strength ring with offset (.input utility = focus:ring-2 focus:ring-ring; shadcn Input = focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2). A 30%-opacity primary ring against #fff is a weak, low-contrast focus indicator and is inconsistent with the rest of the app's keyboard-focus affordance.
  - **Evidence:** LeadForm.tsx:41 (inputClasses '... focus:ring-2 focus:ring-primary/30'), WhitepaperGate.tsx:137,155,173 (same), vs index.css:155 (.input focus:ring-ring) and input.tsx:11 (focus-visible:ring-ring + ring-offset-2)
  - **Fix:** Use the shared <Input> component (as FreeChapterForm already does) or the .input utility class for these forms instead of an ad-hoc string, so focus becomes full-strength focus:ring-2 focus:ring-ring focus:ring-offset-2. At minimum change focus:ring-primary/30 to focus:ring-ring and add ring-offset to meet a visible 3:1 UI focus indicator.
  - **Rule:** _Interactivity and States: Missing/weak focus ring (a11y requirement)_
- **🔴 HIGH · Accessibility** — WhitepaperGate's newsletter opt-in is rendered as a plain <button type='button'> that swaps CheckSquare/Square icons, with no role='checkbox' and no aria-checked. Screen-reader users get an unlabeled toggle button with no announced checked/unchecked state, and there's no real form control behind the opt-in (which is legally meaningful — it governs marketing consent). It also can't be reached/operated as a checkbox by assistive tech.
  - **Evidence:** WhitepaperGate.tsx:180-191 (button with CheckSquare/Square, no role/aria-checked)
  - **Fix:** Use a real checkbox: either the project's shadcn Checkbox + Label, or a native <input type='checkbox' checked={newsletter}> visually styled with the icons. If keeping the button, add role='checkbox' and aria-checked={newsletter} plus an associated label.
  - **Rule:** _Code Quality: Div soup / non-semantic controls (use real form controls); a11y state announcement_
- **🟠 MED · Interactivity & States** — No button in this cluster has an :active/pressed state. .btn-primary (used by LandingPage CTA, LeadForm submit, WhitepaperGate submit/download, CalendlyEmbed, FreeChapterForm download) defines only hover; the shadcn Button used by FreeChapterForm submit also has only hover. Per the checklist every interactive control needs pressed feedback (scale 0.98 or translateY(1px)). On a form-heavy lead-capture surface the submit button is the single most important control and it feels dead on click.
  - **Evidence:** index.css:97-107 (.btn-primary has hover-only, no :active), button.tsx:12 (default variant hover-only), exercised at LeadForm.tsx:318, WhitepaperGate.tsx:195, FreeChapterForm.tsx:96, CalendlyEmbed.tsx:27
  - **Fix:** Add active:scale-[0.98] (or active:translate-y-px) and active:bg-primary/95 to .btn-primary and to the Button default variant. This is a one-line system fix that lights up every CTA in the cluster.
  - **Rule:** _Interactivity and States: No active/pressed feedback_
- **🟠 MED · Strategic Omissions** — Validation is inconsistent across the three forms. LeadForm validates with Zod and renders inline per-field error messages (good). FreeChapterForm and WhitepaperGate rely solely on the native 'required' attribute and email type, so users get inconsistent native browser tooltips rather than the styled inline error pattern, and there is no format/length validation. The checklist calls for client-side validation and inline errors as a baseline; here the same product ships two different validation UXs.
  - **Evidence:** LeadForm.tsx:8-17,82-93,151-153 (Zod + inline errors) vs FreeChapterForm.tsx:75-93 (required-only) and WhitepaperGate.tsx:132-176 (required-only)
  - **Fix:** Apply the LeadForm pattern to the other two: a small Zod schema (name min length, valid email) with inline <p className='text-sm text-destructive'> messages under each field, clearing on change. Keeps validation UX consistent across every lead form.
  - **Rule:** _Strategic Omissions: No form validation_
- **🟠 MED · Interactivity & States** — The hand-rolled inputs in LeadForm and WhitepaperGate set no placeholder text color, so they inherit the browser default placeholder (which varies by browser and is not the design token). The design system's Input sets placeholder:text-muted-foreground explicitly. Result: placeholder color is inconsistent with the rest of the app and unverified for contrast on the near-white bg-background card.
  - **Evidence:** LeadForm.tsx:41 (inputClasses has no placeholder: class), WhitepaperGate.tsx:137,155,173 (same) vs input.tsx:11 (placeholder:text-muted-foreground)
  - **Fix:** Add placeholder:text-muted-foreground to the shared input class (or switch to the <Input> component). muted-foreground is hsl(220 10% 40%), which gives readable, on-brand placeholder contrast.
  - **Rule:** _Color and Surfaces: light placeholder on near-white / consistency_
- **🟠 MED · Component Patterns** — NichePage's CTA section is the textbook generic pattern the checklist flags: exactly one filled button + one secondary/ghost button side by side. Combined with the Title Case heading 'Ready to Start Your Transformation?', this reads as default AI scaffolding. LandingPage's CTA section similarly leans on the same single filled CTA inside a generic glass-card.
  - **Evidence:** NichePage.tsx:240-256 (btn-primary + btn-secondary pair), heading at NichePage.tsx:235 ('Ready to Start Your Transformation?')
  - **Fix:** Differentiate the two actions by hierarchy rather than just fill: keep one prominent button and demote the secondary to a text link (e.g. 'or download the framework'). Convert the heading to sentence case: 'Ready to start your transformation?'.
  - **Rule:** _Component Patterns: Always one filled button + one ghost button; Content: Title Case headers_
- **🟠 MED · Interactivity & States** — NichePage mounts all body content with opacity-0 and reveals it only through JS (useEffect setTimeout + classList add of animate-fade-in/animate-fade-in-up, with hardcoded 100-800ms staggered delays). If JS is disabled, errors before this effect runs, or hydration is delayed, the hero, challenges, solutions, case study and CTA all stay invisible — the page renders blank. The animation is also imperative classList toggling rather than CSS/IntersectionObserver, and is not scroll-driven.
  - **Evidence:** NichePage.tsx:62-103 (setTimeout chain) and the opacity-0 starting classes at NichePage.tsx:117,134,162,189,233
  - **Fix:** Drive entrance animation from CSS (animation on mount) or an IntersectionObserver that only animates when JS is present, and ensure the default (no-JS) state is fully visible — i.e. do not gate base visibility on JS. Prefer a single reusable reveal mechanism over per-element setTimeout offsets.
  - **Rule:** _Interactivity and States: animation should not hide content; Code Quality: fragile JS-gated rendering_
- **🟠 MED · Code Quality** — CalendlyEmbed is misnamed and carries dead-comment cruft. The component is named CalendlyEmbed and the file opens with a commented-out iframe snippet, but it actually links to Microsoft Bookings via BOOKING_URL and renders no embed at all — just a button plus a mailto fallback. The name and leading comment are misleading artifacts that will confuse future maintainers.
  - **Evidence:** CalendlyEmbed.tsx:1-2 (commented iframe + 'Microsoft Bookings' comment), CalendlyEmbed.tsx:8 (component named CalendlyEmbed), CalendlyEmbed.tsx:23-32 (just an <a> link, no embed)
  - **Fix:** Rename the component/file to BookingCTA (or DiscoveryCallCard) to match what it does, and remove the stale commented iframe. If an actual inline embed is desired, implement it; otherwise the 'Embed' name is wrong.
  - **Rule:** _Code Quality: Commented-out dead code; misleading naming_
- **🟡 LOW · Code Quality** — LandingPage imports CheckCircle from lucide-react but never uses it (only ArrowRight and AlertTriangle are referenced). Dead import left from a prior iteration.
  - **Evidence:** LandingPage.tsx:3 (import { ArrowRight, AlertTriangle, CheckCircle }) — CheckCircle unused in the file
  - **Fix:** Remove CheckCircle from the import.
  - **Rule:** _Code Quality: Commented-out / dead code artifacts_
- **🟡 LOW · Component Patterns** — Every section in this cluster is wrapped in the same generic .glass-card (white bg + border + shadow + rounded-2xl). LandingPage stacks four of them (problem, solution, optional testimonial, CTA), NichePage uses it for challenges, solutions, case study and CTA, and all three forms plus CalendlyEmbed sit in one too. The result is a monotonous wall of identical white cards-on-near-white where elevation no longer communicates hierarchy (the checklist's 'card should exist only when elevation communicates hierarchy').
  - **Evidence:** index.css:77-79 (.glass-card), used at LandingPage.tsx:77,96,109,135; NichePage.tsx:134,162,189,233; LeadForm.tsx:120; WhitepaperGate.tsx:53; CalendlyEmbed.tsx:10

  - **Evidence:** PrepPromptCard.tsx:2,112 (Sparkles); ResearchTab.tsx:2,142; ExperimentTab.tsx:2,134; all product files import lucide-react
  - **Fix:** If differentiation is wanted, move the icon set to Phosphor/Heroicons or a small custom set, and swap Sparkles for a less clichéd 'assist' mark (e.g. wand-off-grid, bolt). At minimum avoid Sparkles for anything AI-labelled.
  - **Rule:** _Iconography — Lucide/Feather exclusively; cliché metaphors_
- **🟡 LOW · Layout** — Uniform radii fight each other once nested: outer glass-card is rounded-2xl while every inner row/card and the inputs are rounded-xl/rounded-lg, and chips are fully rounded-full — with no deliberate tighter-inner / softer-outer system, the nesting looks accidental rather than intentional.
  - **Evidence:** ExperimentTab.tsx:152,183 (rounded-2xl / rounded-xl); InterventionCandidatesTable.tsx:75; TestCardList.tsx:64; ReactionChips.tsx:26 (rounded-full); src/index.css:36 (--radius 0.75rem)
  - **Fix:** Define an explicit radius scale (e.g. container rounded-2xl, inner card rounded-xl, control rounded-md) and apply it consistently so the nesting reads as a system, not coincidence.
  - **Rule:** _Layout — uniform/uncoordinated border-radius_

## shadcn/ui primitives (themed look)

> These are stock shadcn/ui primitives wired to the project's HSL token theme. The theming layer is mostly sound: focus-visible rings are present on Button/Input/Textarea/Tabs, primary/secondary/accent contrast all pass WCAG AA, semantics are correct (Table uses real table/thead/tbody, FormControl wires aria-describedby/aria-invalid, DialogClose has sr-only label), and there are no fake-content or copy tells in the primitive layer. The real, code-grounded gaps are: (1) the destructive variant and all form error text fail AA contrast (3.59-3.76:1 white-ish on mid-red); (2) NO variant defines an :active/pressed state anywhere (Button, Badge, TabsTrigger, DialogClose, SelectTrigger all stop at :hover), which the Foundation already flags for .btn-primary and is reproduced here across every interactive primitive; (3) radius is inconsistent both internally (Card rounded-lg=12px vs Button/Input/Select/Dialog rounded-md=10px) and against the project's own components (.glass-card rounded-2xl=16px, .input/.btn-primary rounded-lg), so the primitives never visually agree with the bespoke components; (4) two primitives (Select, Dialog, Badge) use focus: instead of focus-visible:, firing rings on mouse click; (5) Badge is the rounded-full pill cliche, duplicating the project's .tag and .phase-chip. No loading/skeleton primitive exists. Most issues are theming/state gaps rather than stock-structure nitpicks.

**9 findings** — 3 high · 4 medium · 2 low

- **🔴 HIGH · Accessibility** — Destructive variant fails WCAG AA contrast. destructive-foreground (210 40% 98%, near-white) on destructive (0 84.2% 60.2%, mid-red) computes to 3.59:1, below the 4.5:1 body-text minimum. This is the Button variant="destructive" and Badge variant="destructive", used in real delete/error surfaces (Dashboard, NewProject, alert, toast).
  - **Evidence:** button.tsx:13-14 and badge.tsx:15-16 (bg-destructive text-destructive-foreground); tokens in src/index.css:29-30; ratio verified 3.59:1
  - **Fix:** Darken --destructive to roughly 0 72% 45% (raises white-text contrast to ~4.7:1) or keep the red and switch destructive-foreground to pure white plus a subtle darker outline. Either way the destructive label must clear 4.5:1.
  - **Rule:** _Button & form contrast must meet WCAG AA (pre-flight contrast gate)_
- **🔴 HIGH · Accessibility** — Form validation error text fails AA. FormMessage and the FormLabel error state render in text-destructive (0 84.2% 60.2%) directly on the page background (210 20% 98%) = 3.60:1, or on a white card = 3.76:1 — both below 4.5:1 for the small (text-sm) error copy users most need to read. text-destructive is used as error text in LeadForm, FreeChapterForm, Contact, GetTheBook, AssessmentResults and more.
  - **Evidence:** form.tsx:158 (text-sm font-medium text-destructive) and form.tsx:96 (error && text-destructive); ratios 3.60:1 / 3.76:1
  - **Fix:** Introduce a dedicated darker error token for text (e.g. --destructive-text ~ 0 74% 42%, ~5:1 on white) and use it for FormMessage/FormLabel error states, leaving the brighter red for fills only.
  - **Rule:** _Button & form contrast must meet WCAG AA (pre-flight contrast gate)_
- **🔴 HIGH · Interactivity & States** — No :active / pressed feedback on any interactive primitive. Button defines only hover (e.g. hover:bg-primary/90) with no active: state; the same gap repeats on Badge, TabsTrigger, DialogClose and SelectTrigger. Because every product button funnels through this primitive, the missing pressed state is systemic, not a one-off — clicks feel dead with no physical response.
  - **Evidence:** button.tsx:12-20 (variants have hover: only, no active:); badge.tsx:11-17; tabs.tsx:30; dialog.tsx:45; select.tsx:20
  - **Fix:** Add a pressed state to the buttonVariants base and to TabsTrigger/Badge/DialogClose, e.g. active:scale-[0.98] active:bg-primary/80 (and per-variant equivalents). This is the same fix the Foundation calls for on .btn-primary, applied at the primitive source.
  - **Rule:** _Real states: hover AND :active/pressed required_
- **🟠 MED · Component Patterns** — Radius is inconsistent across primitives and disagrees with the project's bespoke components. With --radius=0.75rem, rounded-lg=12px but rounded-md=10px and rounded-sm=8px. Card uses rounded-lg (12px) while Button, Input, Textarea, SelectTrigger, DialogContent and TabsList use rounded-md (10px), and TabsTrigger/SelectItem use rounded-sm (8px). Meanwhile the hand-written components use a different scale entirely: .glass-card is rounded-2xl (hardcoded 16px, not token-derived) and .input/.btn-primary/.btn-secondary are rounded-lg. So a shadcn Card (12px) sits next to a .glass-card (16px) and a shadcn Button (10px) next to a .btn-primary (12px) — corners never match.
  - **Evidence:** card.tsx:12 (rounded-lg) vs button.tsx:8 / input.tsx:11 / select.tsx:20 (rounded-md); src/index.css:78 (.glass-card rounded-2xl) and src/index.css:98,110,155 (rounded-lg); --radius src/index.css:36
  - **Fix:** Pick one radius story. Either bump the primitives' default to rounded-lg so cards/inputs/buttons all read 12px and match .btn-primary, or retune --radius and replace the hardcoded rounded-2xl on .glass-card with a token-derived value so bespoke and shadcn surfaces share a corner radius.
  - **Rule:** _Uniform/!inconsistent border-radius — radius consistency vs --radius_
- **🟠 MED · Interactivity & States** — Inconsistent focus model across primitives: some use focus-visible: (keyboard-only ring) and others use focus: (ring on mouse click too). Button, Input, Textarea, Tabs use focus-visible:ring-2; but Badge, SelectTrigger and DialogClose use plain focus:ring-2 / focus:outline-none. The result is that clicking a Select or the dialog X with a mouse flashes a focus ring while clicking a Button does not — visibly uneven within the same form.
  - **Evidence:** focus-visible: in button.tsx:8, input.tsx:11, textarea.tsx:12, tabs.tsx:30,45; vs focus: in badge.tsx:7, select.tsx:20, dialog.tsx:45
  - **Fix:** Standardize on focus-visible: for the ring across all primitives (Badge, SelectTrigger, DialogClose) so the keyboard focus indicator is consistent and mouse clicks don't show a ring. The project's .input (src/index.css:155) also uses focus: — align it too.
  - **Rule:** _Missing/!consistent focus ring (visible focus indicator)_
- **🟠 MED · Component Patterns** — Badge is the generic rounded-full pill, and it triple-duplicates the project's existing pill patterns. badge.tsx uses rounded-full px-2.5 py-0.5 text-xs; src/index.css already defines .tag (rounded-full px-2.5 py-0.5 text-xs) and .phase-chip (rounded-full). Three near-identical pill systems means status/category labels look interchangeable and the CLEAR-phase chips don't get a distinct shape.
  - **Evidence:** badge.tsx:7 (rounded-full px-2.5 py-0.5 text-xs); src/index.css:138 (.tag rounded-full) and src/index.css:150 (.phase-chip rounded-full)
  - **Fix:** Differentiate by shape/role: give the Badge a squarer radius (rounded-md, tied to --radius) for inline status, reserve the pill only for one role, and consolidate .tag/.phase-chip onto the Badge variants so there is one labeling primitive instead of three.
  - **Rule:** _Pill-shaped New/Beta badges — badge pill cliche_
- **🟠 MED · Interactivity & States** — No loading/skeleton primitive in the shared UI set. The primitive folder ships Button/Card/Input/etc. but no Skeleton, and Button has no loading prop or spinner-vs-skeleton affordance. Forms that submit (LeadForm, FreeChapterForm, Contact) have no primitive to render a layout-shaped loading state, pushing each surface toward ad-hoc spinners.
  - **Evidence:** button.tsx (no isLoading/disabled-spinner path); no skeleton.tsx among the audited primitives in src/components/ui/
  - **Fix:** Add a Skeleton primitive (animate-pulse bg-muted rounded-md) matched to card/input shapes, and an optional loading state on Button that swaps the label for a spinner and sets aria-busy, so submit flows show a skeleton-or-inline-loader instead of nothing.
  - **Rule:** _No loading states — skeleton not spinner_
- **🟡 LOW · Color & Surfaces** — Card is the textbook border+shadow+white surface, identical in spirit to the Foundation's flagged .glass-card. card.tsx is border bg-card shadow-sm with no token tint or hover, so every Card looks the same regardless of whether elevation actually communicates hierarchy.
  - **Evidence:** card.tsx:12 (rounded-lg border bg-card text-card-foreground shadow-sm)
  - **Fix:** Tint the shadow toward the cool background hue (e.g. a 220-hue low-opacity shadow rather than default black) and consider dropping the border when shadow already conveys lift, so Cards earn elevation instead of defaulting to it. Only re-mention because the primitive reproduces the Foundation pattern at source.
  - **Rule:** _Generic card look (border + shadow + white background)_
- **🟡 LOW · Iconography** — Primitives hard-wire Lucide icons, the default AI icon set. Dialog imports X, Select imports Check/ChevronDown/ChevronUp directly from lucide-react inside the primitive, so the close affordance and select chevrons are locked to Lucide with no project-level swap point.
  - **Evidence:** dialog.tsx:3 (import { X } from "lucide-react"); select.tsx:3 (Check, ChevronDown, ChevronUp from lucide-react)
  - **Fix:** If the product wants visual differentiation from stock shadcn, route these through a small shared icon module (or Phosphor/Heroicons) so the close/select glyphs match the rest of the brand's iconography and stroke weight in one place.
  - **Rule:** _Lucide or Feather icons exclusively_
