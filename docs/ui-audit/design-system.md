# Design system & foundation audit

_Files: `src/index.css`, `tailwind.config.ts`, `src/components/Layout.tsx`._ These are **global** findings — fixing them lifts every page at once, so apply this tier first.

> The design system is a stock shadcn/Vite starter lightly themed for the CLEAR brand. Genuinely good: a single locked --primary blue accent threads through nav, CTAs and tags on the marketing pages; motion keyframes (fade-in, fade-in-up, blur-in, rotate-slow) all animate only transform/opacity (GPU-safe); html has scroll-behavior:smooth; there is a thoughtful print stylesheet; --ring is defined and .input wires up a real focus ring; meta/OG tags and a favicon.ico asset exist. The problems are the classic AI-foundation tells. (1) It is Inter-everywhere: the 'display' family is SF Pro Display, which only resolves on Apple hardware and silently falls back to Inter var for headings on every other device, so headings and body are the same typeface with no display character. (2) The CLEAR phase system defines FIVE distinct accent hues (--phase-c/l/e/a/r) that render as five different chip colors and five different card border colors on the Methodology page, directly breaking the one-accent lock. (3) Interactive states are incomplete at the source: .btn-primary/.btn-secondary/.nav-link have hover but no :active/pressed state and no focus-visible ring (only .input has a ring). (4) Dark mode is wired in config (darkMode:['class'], dark: variants used in ui components) but there is NO .dark{} token block in index.css, so dark mode is dead. (5) Surfaces are misnamed 'glass' but are plain white+border+shadow with default black shadows, and the radius vocabulary is inconsistent (rounded-2xl cards, rounded-lg buttons, rounded-full chips) with no real scale. Fixing the font, collapsing the phase palette to tints of one accent, and adding :active + focus-visible at the .btn level would lift every page at once.

**15 findings** — 6 high · 6 medium · 3 low

- **🔴 HIGH · Typography** — Inter-everywhere disguised as a display pairing. The 'display' font stack is ['SF Pro Display','Inter var','sans-serif'] and 'sans' is ['Inter var','sans-serif']. SF Pro Display is a system font that only resolves on Apple devices; on Windows/Android/Linux (and most visitors) every heading falls back to Inter var — the exact same face as body text. So headings have no distinct typeface, and the system reads as the #1 AI tell: Inter for everything.
  - **Evidence:** tailwind.config.ts:68-71 (fontFamily.sans + fontFamily.display); index.css:67-70 (h1-h6 -> font-display); home__desktop-hero.png (heading 'Organizational Change Through Behavioral Design' renders in Inter)
  - **Fix:** Ship a real self-hosted display face with character (e.g. Geist, Outfit, Cabinet Grotesk, or a serif for editorial weight) as the first entry of the display stack instead of relying on SF Pro Display, which only exists on Apple. Keep Inter for body. Verify on a non-Apple device that headings differ from body.
  - **Rule:** _Browser default fonts or Inter everywhere_
- **🔴 HIGH · Color & Surfaces** — Five-color CLEAR phase palette breaks the one-accent lock. --phase-c (blue), --phase-l (teal), --phase-e (green), --phase-a (amber), --phase-r (pink) are five fully distinct hues. On the Methodology page each phase chip and each step-card's left accent bar renders in its own color, producing a five-color rainbow against an otherwise blue-locked product. This is the explicit pre-flight violation called out in the brief.
  - **Evidence:** index.css:47-53 (--phase-c/l/e/a/r definitions); index.css:149-151 (.phase-chip); methodology__desktop-full.png ('The Five Steps' cards show blue/teal/green/amber/pink chips and matching left borders)
  - **Fix:** Collapse the phase identity to ONE accent. Use the locked --primary blue at five tints/shades (e.g. 5 steps of lightness) or 5 neutral steps with a single accent reserved for the active phase, instead of 5 separate hues. If color-coding the phases is a hard product requirement, demote it to small numbered markers and keep the accent monochrome.
  - **Rule:** _More than one accent color / Color-consistency lock (one accent locked across the page)_
- **🔴 HIGH · Interactivity & States** — No :active/pressed state on any system button. .btn-primary and .btn-secondary define only hover transitions; there is no scale(0.98)/translateY(1px) or background shift on press, so every CTA across the site feels unresponsive to clicks. This is a foundation-level miss that affects every page.
  - **Evidence:** index.css:97-107 (.btn-primary, hover-only); index.css:109-111 (.btn-secondary, hover-only)
  - **Fix:** Add active:scale-[0.98] (or active:translate-y-px) plus a slightly darker background on active to both .btn-primary and .btn-secondary, with the existing 200ms transition. Mirror the same pressed feedback on any custom buttons.
  - **Rule:** _No active/pressed feedback_
- **🔴 HIGH · Accessibility** — No visible focus ring on buttons or nav links. Only .input wires up focus:ring-2 focus:ring-ring. .btn-primary, .btn-secondary and .nav-link have no focus-visible styling, so keyboard users get only the (frequently suppressed) default browser outline on the primary interactive controls. A --ring token exists but is unused outside inputs.
  - **Evidence:** index.css:154-156 (.input has focus:ring-2 focus:ring-ring); index.css:81-95 (.nav-link, no focus style); index.css:97-111 (.btn-primary/.btn-secondary, no focus style)
  - **Fix:** Add focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2 to .btn-primary, .btn-secondary and .nav-link (reuse the existing --ring token). This is a WCAG keyboard-navigation requirement, not polish.
  - **Rule:** _Missing focus ring_
- **🔴 HIGH · Color & Surfaces** — Dark mode is configured but undefined, so it is dead/broken. tailwind.config sets darkMode:['class'] and shadcn ui components (alert.tsx, chart.tsx) use dark: variants, but index.css contains only a :root light token block — there is NO .dark{} block remapping --background/--foreground/--card/etc. Toggling the dark class would leave all design tokens at their light values, producing broken low-contrast surfaces.
  - **Evidence:** tailwind.config.ts:6 (darkMode:['class']); index.css:6-54 (:root only, no .dark{} found); src/components/ui/chart.tsx and src/components/ui/alert.tsx reference dark: variants
  - **Fix:** Either add a complete .dark{} token block in @layer base that remaps every CSS variable (background, foreground, card, popover, muted, border, input, primary-foreground, phase accents) to dark-appropriate values, or remove darkMode:['class'] and the dark: variants so the system does not advertise a mode it cannot render.
  - **Rule:** _Random dark sections / dark-mode completeness (real states)_
- **🔴 HIGH · Content** — Em-dash used in UI copy, which the pre-flight gate bans. The hero body copy ends with a literal em-dash ('… change for real — not just on paper.'). Because it is foundation-adjacent hero copy seen on the primary entry page, it sets the tone for the whole site.
  - **Evidence:** home__desktop-hero.png (subhead: '… help organizations change for real — not just on paper.')
  - **Fix:** Replace the em-dash with a period or a comma-led clause: 'help organizations change for real, not just on paper.' Audit other UI strings for em-dashes as part of the same pass.
  - **Rule:** _Em-dash ban in UI copy_
- **🟠 MED · Component Patterns** — 'glass' surfaces are a misnomer and are the generic border+shadow+white card. .glass and .glass-card have NO backdrop-filter/blur — they are bg-white/95 + a white border + a default (black) box-shadow. They are exactly the 'border + shadow + white background' generic card the audit flags, and the naming implies glassmorphism that does not exist.
  - **Evidence:** index.css:73-79 (.glass = bg-white/95 border shadow-sm; .glass-card = bg-white/95 border shadow-lg rounded-2xl); services__desktop-full.png and methodology__desktop-full.png (uniform white shadowed cards)
  - **Fix:** Either deliver real glass (add backdrop-blur, a 1px inner border and a subtle inner shadow) for sticky/overlay chrome, or rename to .panel/.card and differentiate cards by spacing/background tone rather than the default border+shadow combo. Reserve elevation for cards where it actually communicates hierarchy.
  - **Rule:** _Generic card look (border + shadow + white background)_
- **🟠 MED · Color & Surfaces** — Default black box-shadows, untinted. .glass uses shadow-sm and .glass-card uses shadow-lg, both Tailwind defaults built on rgba(0,0,0,...). On the cool blue-gray --background (210 20% 98%) these read as flat gray drop shadows with no relationship to the palette.
  - **Evidence:** index.css:74 (.glass shadow-sm); index.css:78 (.glass-card shadow-lg); home__desktop-hero.png / services__desktop-full.png (neutral black card shadows)
  - **Fix:** Define a tinted shadow token (e.g. a blue-tinted hsl(220 40% 20% / low-alpha)) and apply it to .glass/.glass-card so shadows carry the background hue instead of generic black.
  - **Rule:** _Generic box-shadow / colored tinted shadows_
- **🟠 MED · Component Patterns** — Inconsistent radius vocabulary with no real scale. There is a single --radius (0.75rem) feeding lg/md/sm, but components hardcode unrelated radii: .glass-card is rounded-2xl (1rem), .btn-* and .input are rounded-lg, .tag and .phase-chip are rounded-full. The result is neither a deliberate 'tighter inner / softer outer' system nor uniform — it is ad hoc.
  - **Evidence:** index.css:36 (--radius:0.75rem); index.css:78 (.glass-card rounded-2xl); index.css:98/110/155 (rounded-lg); index.css:138/150 (rounded-full)
  - **Fix:** Define an intentional radius scale (e.g. container 16px, card 12px, control 8px, chip pill) tied to tokens and apply it consistently — softer on outer containers, tighter on inner controls — rather than mixing rounded-2xl, rounded-lg and rounded-full arbitrarily.
  - **Rule:** _Uniform border-radius on everything / vary the radius_
- **🟠 MED · Typography** — Heading scale lacks display presence and there is no body measure cap. .heading-xl tops out at text-6xl font-bold leading-tight tracking-tight, but with Inter as the de-facto display face it reads as generic bold sans. Only 400/500/700 weights are in play (no 600 SemiBold rhythm), and no utility caps paragraph width to ~65ch — hero/body copy stretches wide and centered.
  - **Evidence:** index.css:117-127 (heading-xl/lg/md all font-bold leading-tight); index.css:129-135 (body-lg/body-md have no max-width); home__desktop-hero.png (wide centered subhead)
  - **Fix:** Give the real display font more presence (heavier weight, tighter tracking at large sizes, leading ~1.05-1.1) and introduce a 600 weight for sub-hierarchy. Add a max-w-[65ch] (or prose) constraint to .body-lg/.body-md so paragraphs do not exceed ~65 characters.
  - **Rule:** _Headlines lack presence / Body text too wide / Only Regular and Bold weights used_
- **🟠 MED · Color & Surfaces** — Two gray families mixed. The product palette is built on hue 220 (blue-gray: --background 210, --foreground/--muted/--border all 220), but the sidebar tokens switch to hue 240 (a cooler, slightly violet gray: --sidebar-foreground 240 5.3% 26.1%, --sidebar-accent 240 4.8% 95.9%, etc.). Mixing 220 and 240 gray families is the inconsistent-gray tell.
  - **Evidence:** index.css:9-34 (app tokens on hue 220/210); index.css:38-45 (--sidebar-* on hue 240)
  - **Fix:** Retint the --sidebar-* neutrals to the same hue family as the app (220) so all grays share one temperature, or drop the sidebar token block entirely if no sidebar surface ships.
  - **Rule:** _Mixing warm and cool grays / stick to one gray family_
- **🟠 MED · Component Patterns** — Duplicate-CTA 'one filled + one ghost' pattern baked into the system. The two button primitives are .btn-primary (filled blue) and .btn-secondary (low-contrast bg-secondary ghost). Across pages this surfaces as the cliche pair — e.g. hero shows 'Take the Free Assessment' (filled) beside 'Book a Discovery Call' (ghost) — with no tertiary/text-link option to reduce visual noise, and the secondary's gray-on-near-white is low contrast.
  - **Evidence:** index.css:97-111 (.btn-primary + .btn-secondary only); home__desktop-hero.png ('Take the Free Assessment' filled next to 'Book a Discovery Call' ghost)
  - **Fix:** Add a tertiary/text-link button variant to the system and use it for the lower-priority action so pages are not always one filled + one ghost. Ensure the secondary variant's text/background meets 4.5:1 against the page.
  - **Rule:** _Always one filled button + one ghost button_
- **🟡 LOW · Component Patterns** — Pill badges everywhere. .tag is a rounded-full pill (bg-primary/10 text-primary) and .phase-chip is a full circle; the eyebrow labels ('The CLEAR Change Framework', section tags) render as pills. This is the default 'New/Beta pill' look used as the only label treatment.
  - **Evidence:** index.css:137-139 (.tag rounded-full); index.css:149-151 (.phase-chip rounded-full); home__desktop-hero.png (pill eyebrow above the hero heading)
  - **Fix:** Offer at least one non-pill label treatment (squared badge, underline/kicker, or plain uppercase tracked text) and use it for eyebrows so labels are not uniformly pills.
  - **Rule:** _Pill-shaped New and Beta badges_
- **🟡 LOW · Code Quality** — No favicon link in index.html. A favicon.ico asset exists in /public (served at /favicon.ico by Vite convention) but index.html declares no <link rel='icon'>, no apple-touch-icon, and no SVG/branded icon link, so the favicon is implicit-only and incomplete across platforms.
  - **Evidence:** index.html:4-39 (<head> has no rel=icon/apple-touch-icon); public/favicon.ico exists but is never referenced
  - **Fix:** Add explicit <link rel='icon' href='/favicon.ico' sizes='any'>, an SVG icon, and an apple-touch-icon to <head> so the brand mark renders on all platforms and home-screen pins.
  - **Rule:** _Missing favicon_
- **🟡 LOW · Interactivity & States** — Active nav state is visually identical to hover. .nav-link.active:after and .nav-link:hover:after both set the underline to w-1/2 with the same color/weight, so a hovered non-current link looks the same as the current page; on touch (no hover) the only persistent current-page cue is this thin underline.
  - **Evidence:** index.css:89-95 (.nav-link:hover:after and .nav-link.active:after both -> w-1/2)
  - **Fix:** Differentiate the active state from hover — e.g. give .active a filled/wider underline plus text-foreground (and/or font-medium), and reserve the half-width underline for hover only.
  - **Rule:** _No indication of current page in navigation_
